Time-Travel Querying

Reconstructing historical snapshots, timestamp resolution algorithms, and programmatic PyIceberg/SQL lineage travel.

I. Historical Snapshots & Timeline Lineage

Because Iceberg represents a table's state explicitly as an array of immutable snapshots, we can perform historical queries to audit modifications, rollback catastrophic errors, or run reproducible analytics.

💡 Time Travel: The Plain English Concept
Think of Time Travel like checking out files in a Git repository:
• Every commit in Git represents a snapshot of your project's files. You can type git checkout [commit-id] to instantly restore your workspace to that exact moment in history.
• Iceberg works exactly the same way. Every commit adds a snapshot ID with a timestamp. When you query with Time Travel, you tell the engine: "Reconstruct the table exactly as it looked at commit ID 4529" or "as of yesterday at 5 PM."
Iceberg simply loads that historical snapshot's manifest list, completely bypassing all subsequent updates.

1. Mathematical Formulations of Time Travel Mapping

Let $T$ be the table history containing an ordered sequence of snapshots $T = \langle S_0, S_1, \dots, S_n \rangle$. Each snapshot $S_k$ is annotated with a unique identifier $\text{id}(S_k)$ and a Unix commit timestamp $t(S_k) \in \mathbb{N}$ representing when the snapshot was committed.

The reconstitution function $\Psi(S)$ extracts the active set of rows corresponding to a given snapshot state. Time Travel reconstructs the active dataset $R_{\text{active}}$ under two different query modes:

Definition 1: AS OF Snapshot ID
$$R_{\text{active}}(\text{id}_{\text{target}}) = \bigcup_{d \in D_{\text{target}}} \Psi(d) \quad \text{where} \quad S_{\text{target}} = S_k \text{ s.t. } \text{id}(S_k) = \text{id}_{\text{target}}$$
The Mapping: By specifying a precise snapshot ID, the reader directly loads that snapshot's manifest list $D_{\text{target}}$, skipping the current table root pointer entirely. This guarantees mathematically exact reproducibility.
Definition 2: AS OF Timestamp
$$R_{\text{active}}(t_{\text{target}}) = R_{\text{active}}(\text{id}(S_m)) \quad \text{where} \quad m = \max \{ k \mid t(S_k) \le t_{\text{target}} \}$$
The Mapping: If a user requests state as of a physical clock timestamp $t_{\text{target}}$, the query engine scans the metadata snapshot array to find $S_m$—the latest snapshot committed at or before the targeted timestamp.

2. Snapshot Evolution Tree

The table metadata tracks snapshots sequentially. Old snapshots are not deleted immediately; they remain in cloud storage until explicitly purged by maintenance retention cycles: 

[SNAPSHOTS TIMELINE (Immutable & Auditable)]
 Snapshot 0 (id=101, t=10:00) ──> manifest-list-101.avro ──> data-1.parquet
                                                                 │
                                                                 ▼
 Snapshot 1 (id=102, t=11:00) ──> manifest-list-102.avro ──> data-1.parquet, data-2.parquet
                                                                 │
                                                                 ▼ (Latest Root Pointer)
 Snapshot 2 (id=103, t=12:00) ──> manifest-list-103.avro ──> data-1.parquet, data-2.parquet, data-3.parquet

* Time Travel "AS OF 11:30" resolves to Snapshot 102. It scans data-1 and data-2, completely ignoring data-3!
Lineage SQL & Programmatic Queries
-- 1. Query table state as of a specific historical snapshot ID
SELECT count(*) FROM catalog.db.user_events 
  VERSION AS OF 890213084920;

-- 2. Query table state as of a precise timestamp in the past
SELECT * FROM catalog.db.user_events 
  TIMESTAMP AS OF '2026-05-25 10:00:00';

-- 3. Query historical records using system snapshot pathing syntax
SELECT * FROM catalog.db.user_events.snapshots; -- Inspects table snapshot timeline metadata
# Programmatic time-travel and data scanning using PyIceberg
from pyiceberg.catalog import load_catalog

catalog = load_catalog("rest_catalog")
table = catalog.load_table("db.user_events")

# 1. Scan data as of a specific Snapshot ID
scanner_snap = table.scan(snapshot_id=890213084920)
df_snap = scanner_snap.to_arrow() # Reconstruct PyArrow RecordBatch batch

# 2. Inspect active snapshots list to find a timestamp targets
for snapshot in table.metadata.snapshots:
    print(f"Snapshot ID: {snapshot.snapshot_id} committed at: {snapshot.timestamp_ms}")
