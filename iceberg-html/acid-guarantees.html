I. The ACID Transaction Model

Traditional database engines rely on central lock coordinators to guarantee atomic transactions. In distributed cloud object storage, locks are slow and expensive. Apache Iceberg provides full ACID transactions natively using lock-free optimistic concurrency control (OCC).

💡 ACID in the Lakehouse: The Plain English Concept
Imagine two writers trying to edit the same online document at the same time:
• Pessimistic Locking is like forcing a writer to wait outside a locked room until the first writer finishes. This is slow and throttles performance.
• Optimistic Concurrency Control lets both writers draft their edits in separate, private drafts. When committing:
  1. The first writer to hit "Publish" succeeds instantly.
  2. The second writer gets a notice: "A new version was published. We will merge your drafts. If your edits overlap, you must retry."
This guarantees that no write conflicts go unnoticed, without ever blocking writers with active locks.

1. Mathematical Formulation of OCC Conflict Gating

Let a table commit evolve through a series of metadata snapshots. Let $S_{\text{base}}$ be the initial version at which a transaction begins. Let $\Delta_1$ be the pending changes committed by a concurrent transaction, pushing the table state to $S_{\text{concurrent}}$. Let $\Delta_2$ be our local transaction's changes.

During commit, the engine compares the read and write sets to detect overlaps. A conflict exists if the logical files modified by the concurrent transaction overlap with the files read or written by the local transaction:

Conflict Gating Predicate
$$\text{Conflict}(\Delta_1, \Delta_2) \iff \Big( \text{WriteSet}(\Delta_1) \cap \text{ReadSet}(\Delta_2) \neq \emptyset \Big) \lor \Big( \text{WriteSet}(\Delta_1) \cap \text{WriteSet}(\Delta_2) \neq \emptyset \Big)$$
The Constraint: If the intersection ($\cap$) of the files modified concurrently by Transaction 1 and the files evaluated/written by Transaction 2 is non-empty, a conflict is triggered. The catalog aborts Transaction 2, forcing it to rebuild its delta ($\Delta_2$) on top of $S_{\text{concurrent}}$ and retry. If the intersection is empty, the transactions are mathematically independent and both commit successfully.

2. Formal Proof: The ACID Proof Rules

Iceberg satisfies the formal properties of ACID through specific metadata invariants:

ACID Dimension Physical Metadata Proof Invariant Plain English Translation
Atomicity A snapshot commit is a single pointer atomic swap of the root JSON metadata path: $$TableMetadata_v \xrightarrow{\text{Catalog PUT}} TableMetadata_{v+1}$$ Either the entire snapshot commit succeeds and becomes visible globally, or it fails completely. No partial files or intermediate states are ever exposed.
Consistency All mutations preserve Stable ID constraints: $$\forall id \in \text{Domain}(\Sigma), \quad \text{Type}(id) = \text{stable}$$ The schema's structural types cannot mismatch across readers because Iceberg maps column statistics via stable, immutable Field IDs rather than shifting string column names.
Isolation Supports Snapshot Isolation (SI) and Serializable Snapshot Isolation (SSI) via OCC sequence gating: $$\text{Seq}(d) \le \text{Seq}(f_{\text{pos}})$$ Readers scan a completely isolated, static snapshot tree. Concurrent writes remain invisible until their transaction is atomically swapped and registered.
Durability Physical data files are strictly immutable: $$\text{Content}(f.\text{path}) = \text{constant}$$ Once the catalog swap completes, all data and manifest files are locked as immutable cloud objects, guaranteeing permanent durability across cluster crashes.

3. The Optimistic Concurrency Control (OCC) Workflow

The following flowchart outlines how the Iceberg catalog coordinates lock-free concurrency checks: 

[Transaction Starts] ----> Read current Snapshot ID (e.g. S_base = 10)
                                 │
                                 ▼
                     Draft edits privately (Delta_2)
                                 │
                                 ▼
[Attempt Commit] --------> Query Catalog for latest Snapshot ID (S_latest)
                                 │
                   Is S_latest == S_base? (No Concurrent commits)
                     ├── YES ──> Swap metadata pointer atomically to S_base + 1 (SUCCESS!)
                     └── NO  ──> Read concurrent modifications (Delta_1)
                                 │
                            Check Conflicts:
                            Does Delta_1 intersect with ReadSet/WriteSet of Delta_2?
                              ├── YES ──> [ABORT & FAIL] (Throw CommitFailedException)
                              └── NO  ──> [RESOLVE & RETRY]:
                                          Rebase S_base = S_latest
                                          Re-evaluate Delta_2 against S_latest
                                          Attempt Commit Again!

-- Configure Spark SQL for aggressive commit retries in high-concurrency environments
ALTER TABLE catalog.db.iceberg_table SET TBLPROPERTIES (
  'commit.retry.num-retries' = '10',          -- Retry up to 10 times before failing
  'commit.retry.min-wait-ms' = '100',         -- Backoff at least 100ms
  'commit.retry.max-wait-ms' = '2000',        -- Cap backoff wait at 2 seconds
  'history.expire.max-snapshot-age-ms' = '86400000' -- Keep snapshot history for 24 hours
);
