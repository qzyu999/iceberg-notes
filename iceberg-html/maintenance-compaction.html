Maintenance & Compaction

Understanding structural compaction constraints, snapshot expiration, and storage garbage collection safety proofs.

VI. Compaction & Storage Maintenance

Maintenance pipelines physically rewrite table structures to optimize scan paths and reclaim storage, ensuring zero changes to active logical rows.

1. Data Compaction & Parity

Let $D_{\text{frag}} \subset D_k$ be a subset of fragmented data files targeted for compaction. The compaction operator $\Gamma_{\text{compact}}$ writes a set of consolidated data files $D_{\text{new}}$: $$\Gamma_{\text{compact}}(D_{\text{frag}}) \to D_{\text{new}}$$

Axiomatic Constraint of Compaction
$$\bigcup_{d \in D_{\text{frag}}} \text{ActiveRows}(d, S_k) \equiv \bigcup_{d' \in D_{\text{new}}} \text{ActiveRows}(d', S_k)$$
Compaction must maintain perfect logical row parity while drastically reducing physical file counts ($|D_{\text{new}}| \ll |D_{\text{frag}}|$).
Theorem 3: Compaction Safety and Equivalence
$$\text{ActiveRows}(D_{\text{new}}, S_j) \equiv \text{ActiveRows}(D_{\text{frag}}, S_j) \quad \forall S_j \ge S_{k+1}$$
Theorem 3: Applying compaction $\Gamma_{\text{compact}}$ is a safe metadata transition that does not alter the logical row projection of the table for any future snapshot state.
Proof of Theorem 3:
Let $S_{k+1}$ be the compaction commit snapshot. By the axiomatic constraint of compaction, $\text{ActiveRows}(D_{\text{frag}}, S_k) = \text{ActiveRows}(D_{\text{new}}, S_k)$. Now consider a future snapshot $S_j$ ($j > k$) where a concurrent transaction appends a delete file $f_d$:
  1. Scenario A (Delete committed before Compaction): The delete file $f_d$ has sequence number $s_d \le s_k$. The compaction process reads active rows after applying $f_d$, meaning deleted rows are already filtered out and are not written to $D_{\text{new}}$. Thus, row identity is preserved.
  2. Scenario B (Delete committed after Compaction): The delete file $f_d$ has sequence number $s_d \ge s_{k+1}$. It cannot target $D_{\text{frag}}$ because $D_{\text{frag}}$ is no longer active. The transaction instead writes the delete targeting $D_{\text{new}}$, resolving the active row set correctly.
Thus, the logical row projection remains completely invariant. $\quad \blacksquare$

2. Gated Storage Pruning

Snapshot expiration $\Gamma_{\text{expire}}$ retains a subset of snapshots based on age history $\Delta t_{\text{max}}$: $$\Gamma_{\text{expire}}(T) \to T' = \langle S_m, \dots, S_n \rangle$$ where $m = \max \{ i \mid \text{Time}(S_i) < \text{Now}() - \Delta t_{\text{max}} \}$. Any physical files referenced in expired snapshots $\{ S_0, \dots, S_{m-1} \}$ that are not referenced in any retained snapshot $S \in T'$ are marked for physical deletion.

Theorem 4: Storage Pruning Safety
$$\forall f \in F_{\text{delete}}, \quad \text{Now}() - \text{MTime}(f) > \Delta t_{\text{grace}}$$
Theorem 4: Let $F_{\text{delete}}$ be the set of physical files targeted for physical deletion by snapshot expiration or orphan cleanup. If table GC is enabled and all targeted files have a modification age older than the grace period $\Delta t_{\text{grace}}$ (default 24 hours), then physical deletion is guaranteed to never cause data loss or corrupt concurrent active transactions.
Proof of Theorem 4:
Suppose a file $f \in F_{\text{delete}}$ is deleted and this deletion corrupts an active concurrent transaction $T_{\text{concurrent}}$ committing at $t_{\text{commit}} > \text{Now}()$:
  1. Case 1: $f$ is a data file needed by $T_{\text{concurrent}}$. If $f$ was written by an active concurrent transaction, its modification age is bounded by the transaction duration, which is less than the grace period: $$\text{Now}() - \text{MTime}(f) \le \text{Duration}(T_{\text{concurrent}}) < \Delta t_{\text{grace}}$$ But this contradicts our safety gate constraint: $\text{Now}() - \text{MTime}(f) > \Delta t_{\text{grace}}$. Thus, $f$ could not have been written by any active concurrent transaction.
  2. Case 2: $f$ was active in the catalog but marked for cleanup. For the expiration algorithm to classify $f$ as unreferenced, it must have traversed all active metadata paths starting from the linearizable catalog pointer. Since $f$ is unreferenced in all active snapshots $S \in T'$ and $T_{\text{concurrent}}$ branches from some active state $S_{\text{base}} \in T'$, $T_{\text{concurrent}}$ could never reference $f$ because $f$ was already removed from the metadata tree prior to $S_{\text{base}}$.
Thus, $f$ is completely unreachable by any current or future reader branching from the catalog. Deleting $F_{\text{delete}}$ is perfectly safe. $\quad \blacksquare$
