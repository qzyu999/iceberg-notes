Specification Section I

Table State & Transitions

Formalizing the atomic state progression and transaction protocols of the Apache Iceberg table format.

1. The Root State Pointer

In traditional hive-style tables, a table's state is defined implicitly by the state of directories on a filesystem. In Apache Iceberg, a table's state is defined explicitly by a single metadata JSON file. This metadata file acts as the absolute root pointer of the table format.

Every reader loads this file to obtain:

  • The absolute, canonical table schema (and historical schemas).
  • Active and evolved partition specifications.
  • Active and evolved sort orders.
  • A comprehensive history of table snapshots (which represent historical data states).

2. Atomic State Transitions

Every mutating operation on an Iceberg table (inserting data, deleting rows, evolving partitions, compacting files) is modeled as a formal state transition. Invariants are maintained via optimistic concurrency control (OCC).

Invariant 1.0: Atomic Commit Transition
$$TableMetadata_v \xrightarrow{\Delta} TableMetadata_{v+1}$$
The Invariant: A mutation is successful if and only if the catalog atomically swaps the table's root pointer from version $v$ to the newly generated version $v+1$. If another transaction has committed in the interim, swapping to version $v+2$ from a base of $v$ is rejected by the catalog, forcing the transaction to retry.

During retries, the mutating engine:

  1. Reads the newly committed metadata file ($TableMetadata_{v+1}$).
  2. Checks if the concurrent changes conflict with the pending mutation (e.g., if files were deleted that the mutation relies on).
  3. If no conflicts exist, applies the delta ($\Delta$) against the new state to produce $TableMetadata_{v+2}$ and attempts to commit again.

3. Catalog Protocols

Because filesystems do not generally support atomic lock-free multi-file mutations, Iceberg relies on catalogs to guarantee atomic swaps. The catalog provides a key-value store mapping the table identifier (e.g., db.table_name) to the current metadata JSON path.

Three primary catalog locking mechanisms guarantee Invariant 1.0:

  • Database Transactions (e.g., JDBC catalog updating a relational table row).
  • Atomic Conditional Puts (e.g., DynamoDB catalog using conditional updates, or REST Catalogs using POST deltas).
  • Locking Services (e.g., ZooKeeper or Hive Metastore locks).
