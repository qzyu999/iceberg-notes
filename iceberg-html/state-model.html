Formal State Model

Understanding the Apache Iceberg state machine through formal sets, structural tuples, and algebraic invariants.

I. The Mathematical Model & Core Axioms

To construct a logically sound data lakehouse, the Apache Iceberg format defines the table structure and its physical files as an immutable state machine governed by algebraic invariants rather than transient database catalogs.

Let $\mathcal{U}_{\text{URI}}$ be the universe of valid Uniform Resource Indicators, and $\mathcal{U}_{\text{ID}} \subset \mathbb{N}$ be the universe of unique Field Identifiers. The structural foundation of the table is defined as follows:

Definition 1: Schema
$$\Sigma \subset \{ (id, name, type, required) \mid id \in \mathcal{U}_{\text{ID}}, name \in \text{String}, type \in \text{Type}, required \in \mathbb{B} \}$$
A table schema $\Sigma$ is a directed acyclic structure mapping stable, unique field identifiers to specific physical typed attributes.
Definition 2: Partition Specification
$$P = \{ (source\_id, field\_id, \tau, name) \}$$
A partition spec $P$ of schema $\Sigma$ is a set of transform mappings, where $source\_id \in \text{Domain}(\Sigma)$, $field\_id \in \mathcal{U}_{\text{ID}}$, $\tau: \text{Type}(source\_id) \to \text{PartitionType}$ is a deterministic transformation function, and $name \in \text{String}$.
Definition 3: Physical File
$$f = (path, type, spec\_id, partition, stats)$$
A physical file $f$ is a 5-tuple where $path \in \mathcal{U}_{\text{URI}}$ is the unique storage locator, $type \in \{ \text{Data}, \text{EqualityDelete}, \text{PositionDelete}, \text{DeletionVector} \}$, $spec\_id \in \mathbb{N}_0$ is the active spec ID, $partition$ represents the partition values, and $stats$ tracks null bounds and min/max metrics.
Definition 4: Manifest Entry
$$e = (status, sequence\_number, file\_sequence\_number, file)$$
A manifest entry $e$ acts as the metadata envelope tracking whether a file is $\text{Existing}$, $\text{Added}$, or $\text{Deleted}$, annotated with transaction sequence numbers.
Definition 5: Table State Snapshot
$$S_k = (D_k, F_{e, k}, F_{p, k}, V_{d, k})$$
A snapshot $S_k$ representing commit version $k$ is a 4-tuple consisting of the active set of Data Files ($D_k$), Equality Delete Files ($F_{e, k}$), Position Delete Files ($F_{p, k}$), and Deletion Vector Puffin blobs ($V_{d, k}$).
Definition 6: The Table History
$$T = \langle S_0, S_1, S_2, \dots, S_n \rangle$$
An Iceberg table $T$ is an ordered, append-only sequence of immutable state snapshots driven by atomic metadata catalog swaps.

Core Algebraic Invariants

To ensure correct distributed processing, the table state transitions are governed by three core axioms:

  • Axiom 1: Physical File Immutability — Once written to storage, a physical file is mathematically immutable: $$\forall f \in S_i, \quad \text{Content}(f.\text{path}, t) = \text{Content}(f.\text{path}, t + \delta)$$
  • Axiom 2: Monotonicity of Sequence Numbers — The transaction sequence number strictly increases across snapshots: $$\forall S_i, S_j \in T, \quad i < j \implies \text{Seq}(S_i) < \text{Seq}(S_j)$$
  • Axiom 3: Field ID Uniqueness & Lifecycle Stability — Under all schema evolution mappings $\Sigma \to \Sigma'$, field IDs remain permanently stable: $$\forall id \in \text{Domain}(\Sigma) \cap \text{Domain}(\Sigma'), \quad \text{Type}(id, \Sigma) = \text{Type}(id, \Sigma')$$

💡 Concept Mapping & Plain English Mappings

Symbol / Set Formal Concept Name Plain English Translation
$T$ The Table History A timeline or movie reel of all snapshots from table creation to the latest metadata pointer.
$S_k$ Table State Snapshot A single static capture frame of the table. A JSON/Avro tree tracking all active files.
$D_k$ Active Data Files The actual Parquet files stored in cloud object storage that hold your row records.
$F_{e, k}$ / $F_{p, k}$ Delete Log Files Metadata rule lists specifying row drops (equality conditions or exact line numbers).
$V_{d, k}$ Deletion Vectors Highly optimized Roaring Bitmaps flagging exactly which physical row indexes are logically deleted.
