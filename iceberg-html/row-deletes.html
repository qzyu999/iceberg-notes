Specification Section III

Row-Level Deletes: COW & MOR

Analyzing the mechanics, structural files, and performance tradeoffs of Copy-On-Write and Merge-On-Read patterns in the V2 table specification.

1. The Two Paradigms

Apache Iceberg's Format Version 2 introduces native support for row-level deletes. Prior to V2, deleting a row required rewriting the entire data file containing that row. V2 offers two distinct write-time strategies:

Copy-On-Write (COW)
COW applies deletes immediately during writes by reading the original data file, filtering out the deleted rows, and writing a completely new data file.
  • Write Cost: High (rewriting whole parquet files).
  • Read Cost: Low (no runtime overhead, direct scans).
  • Best for read-heavy tables with infrequent updates.
Merge-On-Read (MOR)
MOR avoids rewriting data files during ingestion. Instead, it writes deletes immediately to separate, highly compressed metadata files called Delete Files.
  • Write Cost: Low (extremely fast append-only logs).
  • Read Cost: Higher (requires joining data and deletes).
  • Best for high-frequency write pipelines and low-latency updates.

2. Delete File Formats & Invariants

In Merge-on-Read, deletes are encoded using two fundamentally different schemas. The engine decides which format to write based on the incoming delete transaction context.

Position Delete Files

A Position Delete File identifies deleted rows by the absolute path of their containing data file and their zero-indexed row number within that file.

Invariant 1.2: Position Delete Pruning
$$\text{Active}(r, F) \iff (F.\text{path}, r.\text{pos}) \notin D_{\text{pos}}$$
The Invariant: A reader determines if a row $r$ at index $pos$ inside a data file $F$ is active by checking if the tuple $(F.\text{path}, r.\text{pos})$ is present in the global set of active position deletes $D_{\text{pos}}$. Because position deletes specify the exact file and index, engines load them into set structures or bit arrays (e.g. Roaring Bitmaps) for $O(1)$ lookup time during the file scan.

Equality Delete Files

An Equality Delete File identifies deleted rows by the value of one or more specified columns. It contains a list of deleted values, and any row in the table containing a matching value in those columns is considered deleted.

Invariant 1.3: Equality Delete Matching
$$\text{Active}(r, F) \iff \forall d \in D_{\text{eq}}, \exists i \text{ s.t. } r.\text{val}_i \neq d.\text{val}_i$$
The Invariant: A row $r$ is active if and only if it does not match any equality filter tuple $d$ in the active set of equality deletes $D_{\text{eq}}$. This operation is structurally more expensive because the reader must evaluate column value equality rules across all files. It requires building a hash join or sorting records, making scan operations $O(N)$ relative to delete volume.
