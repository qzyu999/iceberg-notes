Specification Section II

Partition Spec Evolution

Understanding partition metadata schemas, historical specs, and the grouping mechanics during table compaction.

1. Specification Overview

Iceberg support for **partition spec evolution** allows tables to change how data is physically partitioned over time without rewriting historical files. When a table's partition spec is changed, old data files remain partitioned under the original spec, and only newly written files utilize the evolved spec.

This results in a single snapshot containing files written under multiple different specs simultaneously. Data files carry their specific spec_id, letting readers dynamically prune them.

2. Compaction Planning & Grouping Invariants

When compacting (rewriting) table files, engines must resolve how to group files from old specifications. If old files are rewritten, they must be restructured to fit the current active specification.

Invariant 1.1: Spec Compatibility Grouping
$$G(f) = \begin{cases} f.\text{partition} & \text{if } f.\text{spec\_id} = Table.\text{default\_spec\_id} \\ \emptyset & \text{otherwise} \end{cases}$$
The Invariant: During compaction planning, any file $f$ whose spec_id does not match the active default_spec_id is stripped of its partition value and mapped to the empty record sentinel ($\emptyset$ or emptyStruct). This forces all incompatible old files to be grouped together. The compaction engine will read them in bulk, and the writer will re-partition the entire dataset under the new current specification on write.
