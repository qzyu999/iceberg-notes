Cookbook Recipe 10

Column Statistics & Pruning

Understanding manifest-level column metadata, formal file-skipping mathematical logic, and Puffin-file index sketches.

I. Manifest-Level Statistics & Pruning

In traditional data lakes, query engines must open and parse every data file's footer (e.g. Parquet footers) to read basic statistics, which triggers massive network I/O overhead. Apache Iceberg solves this by hoisting vital column-level statistics directly into the binary Manifest Files (Avro). This allows query planners to skip millions of irrelevant data files instantly without making single object storage reads.

💡 Table Statistics: The Library Card Catalog Analogy
Imagine searching a physical library for books published strictly between 1920 and 1930:
• Traditional Data Lakes are like forcing the librarian to walk to the shelves, pick up every single book, open its cover, and check the publication page. This takes hours.
• Iceberg Table Stats are like checking a drawer-based Card Catalog at the front desk. The card for each bookcase shelf states: "Shelf 12 holds books from years 1950 to 1980". You instantly know Shelf 12 contains nothing for you, so you completely skip walking to it!
Iceberg manifests hold cards containing the minimum, maximum, and null counts for every column, letting readers skip entire physical files without looking inside.

1. The Manifest Entry Statistics Schema

Every manifest entry record contains a data_file struct that registers these detailed stats for each column:

  • column_sizes: Map from Field ID to physical column size in bytes (essential for cost-based query planning).
  • value_counts: Map from Field ID to total values written (including nulls and NaNs).
  • null_value_counts: Map from Field ID to count of null values.
  • nan_value_counts: Map from Field ID to count of NaN (Not-a-Number) values (highly vital for float/double columns).
  • lower_bounds: Map from Field ID to binary serialized minimum values.
  • upper_bounds: Map from Field ID to binary serialized maximum values.

2. Formal Mathematical File Pruning Proofs

Let $f$ be a candidate data file with total records $R_f = f.\text{record\_count}$. Let $c$ be a column corresponding to stable Field ID $i$. The manifest metadata exposes:
  • Lower bound value: $L_i(f)$
  • Upper bound value: $U_i(f)$
  • Null records count: $N_i(f) \in [0, R_f]$

During query planning, the reader compiles the query's logical filter predicate into a file-skipping check. We prove mathematically when a file $f$ is pruned:

Proof Rule 1: Constant Comparison Pruning ($c > K$)
$$\text{Prune}(f \mid c > K) \iff \Big( U_i(f) \le K \Big) \lor \Big( N_i(f) = R_f \Big)$$
In Plain English: Skip file $f$ if the absolute maximum value stored in the file ($U_i(f)$) is less than or equal to $K$ (meaning no row value can possibly exceed $K$), OR if the entire column is made up of null values (meaning no non-null row exists to satisfy the comparison).
Proof Rule 2: Constant Comparison Pruning ($c < K$)
$$\text{Prune}(f \mid c < K) \iff \Big( L_i(f) \ge K \Big) \lor \Big( N_i(f) = R_f \Big)$$
In Plain English: Skip file $f$ if the absolute minimum value stored in the file ($L_i(f)$) is greater than or equal to $K$ (meaning every single row value is at least $K$), OR if the entire column is completely null.
Proof Rule 3: Exact Equality Pruning ($c = K$)
$$\text{Prune}(f \mid c = K) \iff \Big( K < L_i(f) \Big) \lor \Big( K > U_i(f) \Big) \lor \Big( N_i(f) = R_f \Big)$$
In Plain English: Skip file $f$ if the target search key $K$ falls strictly outside the range of values stored in the file (either strictly below the minimum $L_i(f)$ or strictly above the maximum $U_i(f)$), OR if the entire column contains only null values.
Proof Rule 4: Null Check Pruning (c IS NULL)
$$\text{Prune}(f \mid c \text{ IS NULL}) \iff \Big( N_i(f) = 0 \Big)$$
In Plain English: Skip file $f$ if the count of null values in the column ($N_i(f)$) is exactly zero (meaning the column only contains fully populated, non-null values).
Proof Rule 5: Non-Null Check Pruning (c IS NOT NULL)
$$\text{Prune}(f \mid c \text{ IS NOT NULL}) \iff \Big( N_i(f) = R_f \Big)$$
In Plain English: Skip file $f$ if the count of null values in the column ($N_i(f)$) is exactly equal to the total record count ($R_f$) of the file (meaning the column is entirely null).

II. Puffin Files: Offloading Heavy Statistics

While min/max boundaries and null counts are small enough to fit inside manifest records, advanced statistics—such as Bloom Filters (for set containment checks) and Theta Sketches (for counting unique values)—are too massive. Storing them inside manifests would cause metadata bloat. Iceberg solves this by introducing the **Puffin File Specification**, storing heavy statistical structures in separate, highly compressed auxiliary files.

1. Mathematical Foundations of sketches

A. Theta Sketches (Counting Distinct Values - NDV)

To estimate the Number of Distinct Values ($\text{NDV}$) in a massive column without scanning all values, Theta Sketches keep a dynamically binned sample of hash values. Let $U$ be the set of unique values mapped to uniform hashes $h(x) \in [0, 1)$. The sketch retains only hashes less than a threshold parameter $\theta \in (0, 1]$. Let $S = \{ h(x) < \theta \}$ be the binned sample of size $k$.

The cardinality of unique values is estimated as: $$\widehat{\text{NDV}} = \frac{k}{\theta}$$

As the number of unique records increases, the threshold $\theta$ shrinks dynamically to keep the sample size $k$ fixed, providing an extremely high-accuracy estimate with a tiny $O(k)$ memory footprint.

B. Bloom Filters (Set Membership)

A Bloom Filter is a space-efficient probabilistic data structure used to test whether an element is a member of a set. It uses a bitset of $m$ bits and $k$ independent hash functions. The probability of a false positive ($p$) after inserting $n$ elements into a filter of $m$ bits using $k$ optimal hash functions is: $$p \approx \left(1 - e^{-kn/m}\right)^k$$

To minimize the false-positive rate for a targeted size $n$, the optimum number of hash functions is: $$k = \frac{m}{n} \ln 2$$

2. Puffin File Structural Layout

A **Puffin file** consists of a serialized header, followed by multiple independent payload blocks (holding individual Bloom Filters or Theta Sketches), and a trailing index footer:

# Programmatic inspection of manifest-level column statistics using PyIceberg
from pyiceberg.catalog import load_catalog

catalog = load_catalog("rest_catalog")
table = catalog.load_table("db.user_events")

# Inspect the latest snapshot manifest entries to read column bounds
snapshot = table.current_snapshot()
manifests = snapshot.manifests(table.io)

for manifest in manifests:
    for entry in manifest.fetch_entries(table.io):
        data_file = entry.data_file
        print(f"File: {data_file.file_path}")
        print(f"  Total Rows: {data_file.record_count}")
        print(f"  Null Counts: {data_file.null_value_counts}")
        print(f"  Lower Bounds: {data_file.lower_bounds}")
        print(f"  Upper Bounds: {data_file.upper_bounds}")
