Cookbook Recipe 8

Hidden Partitioning

decoupling queries from physical layout via mathematical projection, partition transforms, and automatic predicate pushdown.

I. Partitioning Evolution & Hidden Pruning

In legacy Hive tables, partitioning is user-facing. To query efficiently, a user must know the exact physical partition columns (e.g. WHERE partition_date = '2026-05-25'). If they filter by raw timestamp instead, the query planner fails to prune directories, causing massive full-table scans. Iceberg solves this by decoupling logical queries from physical layout using Hidden Partitioning.

💡 Hidden Partitioning: The Plain English Concept
Imagine sorting mail in a post office:
• Hive Partitioning is like forcing customers to write the exact mailroom shelf ID on their envelopes: "Send to Bob (Shelf 42B)". If the customer forgets the shelf ID, the mail carrier has to search through millions of shelves manually.
• Iceberg Hidden Partitioning lets customers write standard addresses: "Send to Bob, living in Seattle". The post office itself contains a mathematical mapping (zip code transform) that automatically directs the letter to Shelf 42B.
Users write logical filters (e.g. "events today"), and Iceberg translates them to physical files under the hood, completely transparently.

1. Mathematical Transforms and Specifications

Iceberg partition columns are defined as deterministic transform functions applied to source column values. The V2 specification defines several vital mathematical transform functions:

Transform Function Mathematical Mapping Plain English Operation
identity $$\tau(x) = x$$ Keeps the exact original value (no transform).
bucket[W] $$\tau(x) = (\text{Murmur3Hash}(x) \land \text{0x7FFFFFFF}) \pmod W$$ Hashes the value and bins it into $W$ partitions, ensuring optimal write distribution.
truncate[W] $$\tau(x) = x - (x \bmod W)$$ Truncates numeric values to W-multiples or strings to W-length prefixes.
days $$\tau(x) = \lfloor x / 86,400,000 \rfloor$$ Converts epoch milliseconds (timestamps) to epoch days.
hours $$\tau(x) = \lfloor x / 3,600,000 \rfloor$$ Converts epoch milliseconds (timestamps) to epoch hours.

2. Formal Proof: Scan Pruning by Predicate Pushdown

To prove that queries filtering by raw columns prune partition files without user-facing partition arguments, let us define the Projection Operator ($\Pi$).

Let $c$ be a column of table $T$ with domain $D_c$. Let a query $Q$ contain a filter predicate $F_Q$ defining a subset of valid column values $D_Q \subseteq D_c$. Let a partition spec map column $c$ to a partition field $p$ using the transform function $\tau: D_c \to D_p$.

The query engine computes the partition-level filter bounds by projecting the logical domain $D_Q$ through the transform: $$D_p = \tau(D_Q) = \{ \tau(x) \mid x \in D_Q \}$$

For any partition file $f$ storing records from partition value $v_f \in D_p$, the file is scanned if and only if its partition value intersects with the projected query boundaries: $$\text{Scan}(f) \iff v_f \in \tau(D_Q)$$

Conversely, the scan planner is mathematically guaranteed to prune file $f$ if: $$v_f \notin \tau(D_Q)$$

⚖️ Mathematical Proof Walkthrough: Timestamp to Hour Pruning
Let table $T$ be partitioned by event timestamp transformed to hours: $\tau(t) = \lfloor t / 3,600,000 \rfloor$.
• The user queries: WHERE event_time >= '2026-05-25 12:00:00' AND event_time <= '2026-05-25 13:30:00'.
• In epoch milliseconds, the query bounds correspond to the logical range: $D_Q = [1779710400000, 1779715800000]$.
• Projecting the range through $\tau$: $$\tau(1779710400000) = \lfloor 1779710400000 / 3,600,000 \rfloor = 494364$$ $$\tau(1779715800000) = \lfloor 1779715800000 / 3,600,000 \rfloor = 494365$$
• The projected partition domain is $D_p = [494364, 494365]$ (representing hours 12 and 13).
• The scan engine only reads partition files where $v_f \in \{494364, 494365\}$. Any partition file with $v_f = 494363$ (Hour 11) is pruned, yielding perfect sub-second execution without user-facing partition fields!

3. Physical Mappings: Hive vs. Iceberg

Hidden partitioning completely decouples the physical cloud object folders from table semantics. Hive requires files to reside in rigid folders. Iceberg uses the manifest index to point to files directly, regardless of folder structures: 

[TRADITIONAL HIVE PATHS (Rigid & Fragile)]
 └── s3://bucket/table/
      ├── event_date=2026-05-25/         <-- If renamed or moved, the state breaks
      │    └── data.parquet
      └── event_date=2026-05-26/
           └── data.parquet

[ICEBERG METADATA-DRIVEN INDEX (Decoupled & Robust)]
 ├── manifest.avro  -------> Pointer A: s3://bucket/data/random_hash_A.parquet (Partition: event_date=2026-05-25)
 └── data/                  Pointer B: s3://bucket/data/random_hash_B.parquet (Partition: event_date=2026-05-26)
      ├── random_hash_A.parquet
      └── random_hash_B.parquet
Table Partitioning Schemes
-- Create an Iceberg table with dynamic hidden partitioning transforms in Spark SQL
CREATE TABLE catalog.db.user_events (
  id BIGINT,
  event_time TIMESTAMP,
  category STRING
)
USING iceberg
PARTITIONED BY (
  hours(event_time),                -- Automatically partition by hour internally
  bucket(16, id)                    -- Hash partition IDs across 16 buckets
);
-- Evolve table partitioning dynamically from days to hours without rewriting old files
ALTER TABLE catalog.db.user_events 
  ADD PARTITION FIELD hours(event_time);

-- Old records remain partitioned by day; new records are partitioned by hour.
-- The query planner automatically handles both partition schemes in a single scan!
