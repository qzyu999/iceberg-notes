IV. Copy-on-Write vs. Merge-on-Read Performance Mechanics

Iceberg allows administrators to tune tables precisely between read-optimized Copy-on-Write and write-optimized Merge-on-Read performance profiles.

1. Write/Read Amplification Equations

Let $D_{\text{target}}$ be the set of data files targeted for mutations, $R_{\text{mut}}$ be the records physically modified, and $R_{\text{total}}$ be the total rows in $D_{\text{target}}$.

Copy-on-Write Metrics
$$\text{WAF}_{\text{CoW}} = \frac{\sum_{d \in D_{\text{target}}} \text{SizeOf}(d)}{\text{SizeOf}(R_{\text{mut}})} \approx \frac{\text{SizeOf}(R_{\text{total}})}{\text{SizeOf}(R_{\text{mut}})} \quad , \quad \text{RAF}_{\text{CoW}} = 1.0$$
COW forces an eager rewrite of the entire target data file to apply modifications. This leads to extremely high Write Amplification ($\text{WAF}_{\text{CoW}}$) but keeps Read Amplification ($\text{RAF}_{\text{CoW}}$) at a perfect $1.0$.
Merge-on-Read Metrics
$$\text{WAF}_{\text{MoR}} = \frac{\text{SizeOf}(F_{\text{del}})}{\text{SizeOf}(R_{\text{mut}})} \approx O(1) \quad , \quad \text{RAF}_{\text{MoR}} = \frac{\sum_{d \in D} \text{I/O}(d) + \sum_{f \in F_{\text{del}}} \text{I/O}(f)}{\sum_{d \in D} \text{I/O}(d)} > 1.0$$
MOR appends deletes as a small metadata log file ($F_{\text{del}}$), leaving original data files untouched. This yields near-zero write amplification but increases the read amplification penalty ($\text{RAF}_{\text{MoR}}$) during multi-file scans.

2. Shannon Entropy Limits & Deletion Vector Roaring Containers

To minimize network RTT and object storage space, Deletion Vectors compress row offsets using **Roaring Bitmaps**. Under Shannon's information theory, the minimum average number of bits required to represent $M$ deleted offsets in a file of $N$ rows is bounded by the entropy: $$H_{\text{opt}} \approx \log_2 \binom{N}{M}$$

Roaring Bitmaps compress offsets dynamically into $2^{16}$ row-chunk containers:

  • Array Container: Used when $M_{\text{chunk}} < 4096$. Stores sorted 16-bit integers ($16\text{ bits/offset}$).
  • Bitmap Container: Used when $M_{\text{chunk}} \ge 4096$. Stores a dense bit array of $2^{16}$ bits ($65536\text{ bits}$ total, yielding $\le 16\text{ bits/offset}$).
  • Run-Length Encoding (RLE) Container: Stores contiguous runs as 16-bit pairs of (start_offset, run_length) ($32\text{ bits/run}$), compressing bulk deletions down to a few bytes.
Table Mode Settings (COW vs. MOR)
# Table Properties for strict Copy-On-Write enforcement
write.delete.mode = copy-on-write       # Delete rows by rewriting parquet files
write.update.mode = copy-on-write       # Update rows by rewriting parquet files
write.merge.mode = copy-on-write        # MERGE operations rewrite whole files
# Table Properties for high-throughput Merge-On-Read
write.delete.mode = merge-on-read        # Delete writes immediate V2 Delete Files
write.update.mode = merge-on-read        # Update writes positional/equality deletes
write.merge.mode = merge-on-read         # MERGE writes appends instead of rewrites

# Specific Delete Vector Formats
write.delete.format = positional         # Positional deletes (fast scanning)
# write.delete.format = equality         # Equality deletes (fast writing)
