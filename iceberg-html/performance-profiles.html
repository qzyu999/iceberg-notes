IV. Copy-on-Write vs. Merge-on-Read Performance Mechanics

Iceberg allows administrators to tune tables precisely between read-optimized Copy-on-Write (CoW) and write-optimized Merge-on-Read (MoR) performance profiles. The fundamental tradeoff lies in shifting the compute/IO penalty of mutations from write-time to query-time.

1. Mathematical Formulations of CoW and MoR

Let $D_{\text{target}}$ be the set of data files targeted for mutations, $R_{\text{mut}}$ be the set of records physically modified, and $R_{\text{total}}$ be the total set of rows residing in $D_{\text{target}}$.

Copy-on-Write Metrics
$$\text{WAF}_{\text{CoW}} = \frac{\sum_{d \in D_{\text{target}}} \text{SizeOf}(d)}{\text{SizeOf}(R_{\text{mut}})} \approx \frac{\text{SizeOf}(R_{\text{total}})}{\text{SizeOf}(R_{\text{mut}})} \quad , \quad \text{RAF}_{\text{CoW}} = 1.0$$
Write Amplification Factor ($\text{WAF}$): In CoW, any row update or deletion forces an eager rewrite of the entire target data file containing that row. Thus, $\text{WAF}_{\text{CoW}}$ is proportional to the average file size divided by the mutated record size. For a 128 MB file where 1 row is mutated, the amplification is $O(10^6)$.
Read Amplification Factor ($\text{RAF}$): Because all modifications are fully merged into the physical files at commit time, reading requires no runtime joining. Hence, $\text{RAF}_{\text{CoW}}$ remains at a perfect $1.0$.
Merge-on-Read Metrics
$$\text{WAF}_{\text{MoR}} = \frac{\text{SizeOf}(F_{\text{del}})}{\text{SizeOf}(R_{\text{mut}})} \approx O(1) \quad , \quad \text{RAF}_{\text{MoR}} = \frac{\sum_{d \in D} \text{I/O}(d) + \sum_{f \in F_{\text{del}}} \text{I/O}(f)}{\sum_{d \in D} \text{I/O}(d)} > 1.0$$
Write Amplification Factor ($\text{WAF}$): In MoR, mutations are appended to separate delta log files ($F_{\text{del}}$), leaving original data files completely untouched. This reduces $\text{WAF}_{\text{MoR}}$ to near-zero ($O(1)$ relative to write payload size).
Read Amplification Factor ($\text{RAF}$): Queries must dynamically merge the base data files ($D$) and active delete files ($F_{\text{del}}$), performing run-time reconciliation. This induces a CPU and IO read amplification penalty ($\text{RAF}_{\text{MoR}} > 1.0$), making reads slower.

2. Format V2 Specification & Manifest Structural Evolution

Format Version 1 (V1) tables only support Copy-on-Write mutations, as all manifest entries reference only **Data Files**. Format Version 2 (V2) introduces native Merge-on-Read through the addition of **Delete Files** ($F_{\text{del}}$). To distinguish these file types, the V2 manifest entry schema evolves the nested data_file struct by introducing the content field:

Content Value Enumerated Name Description Format Compatibility
0 DATA Physical data file containing active user records. V1 & V2 Specs
1 POSITION DELETES File identifying deleted rows by file path and relative row offset. V2 Spec Only
2 EQUALITY DELETES File identifying deleted rows by the values of one or more primary keys. V2 Spec Only

Additionally, V2 manifests add two vital metadata fields to each entry to govern correct state tracking:

  • sequence_number: A table-level monotonically increasing sequence tracking the logical time when the data or delete file was committed.
  • file_sequence_number: The sequence number of the commit that added the file to the table metadata.

C. Reconstitution Operator & Transactional Collapse Proof

To prove that this distributed collection of physical files collapsing under a simple scan reconstructs the exact mathematically correct data product, let us formally define the **Reconstitution Operator ($\Psi$)**.

Let the active physical layout in snapshot $S_k$ consist of data files $D_k$, position delete files $F_{p, k}$, and equality delete files $F_{e, k}$. Every data file $d \in D_k$ is a set of physical rows $d = \{ r = (d.\text{path}, i, \text{val}_r) \}$, where $i \in [0, \text{record\_count} - 1]$ is the absolute row offset, and $\text{val}_r$ is the record's values.

The final logical active dataset $R_{\text{active}}$ is defined as the union of the collapse function applied to each data file: $$R_{\text{active}} = \bigcup_{d \in D_k} \Psi(d)$$

The reconstitution operator $\Psi(d)$ filters out deleted rows from data file $d$ by applying the sequence-gated position deletes $F_{p, k}(d)$ and equality deletes $F_{e, k}(d)$: $$\Psi(d) = \{ r \in d \mid \text{PassesPosDeletes}(r, F_{p, k}(d)) \land \text{PassesEqDeletes}(r, F_{e, k}(d)) \}$$

Where the gated sets are bounded strictly by the sequence number invariants: $$F_{p, k}(d) = \{ f \in F_{p, k} \mid \text{Seq}(d) \le \text{Seq}(f) \}$$ $$F_{e, k}(d) = \{ f \in F_{e, k} \mid \text{Seq}(d) < \text{Seq}(f) \}$$

Why does position gating use $\le$ while equality gating uses strictly $<$?
• Position Deletes ($\text{Seq}(d) \le \text{Seq}(f_{\text{pos}})$)
Position deletes refer to precise physical file locations (file path and row index). Because they are tied to a specific file, it is impossible for them to delete rows in files that don't exist yet. They can be safely written in the same transaction that created the data file (e.g. during a Spark MERGE INTO commit), meaning their sequence numbers can be equal ($\text{Seq}(d) = \text{Seq}(f_{\text{pos}}$).
• Equality Deletes ($\text{Seq}(d) < \text{Seq}(f_{\text{eq}})$)
Equality deletes are global, value-based delete rules (e.g., "delete any row where id = 2"). They do not target a specific file. In streaming pipelines (e.g. Flink), a single commit transaction often writes both a new data file (containing id=2) and an equality delete (deleting id=2). If equal sequence numbers were allowed, the delete rule would immediately suppress the newly inserted row in the same transaction! The strict inequality ($\text{Seq}(d) < \text{Seq}(f_{\text{eq}}$) ensures that equality deletes only apply to data files committed before the delete transaction was committed.

And the filtering predicates are defined as: $$\text{PassesPosDeletes}(r, F_{p, k}(d)) \iff \forall f \in F_{p, k}(d), \quad (r.\text{path}, r.\text{pos}) \notin \text{ReadPosDeletes}(f)$$ $$\text{PassesEqDeletes}(r, F_{e, k}(d)) \iff \forall f \in F_{e, k}(d), \forall v_{\text{del}} \in \text{ReadEqDeletes}(f), \exists c \in f.\text{equality\_ids} \text{ s.t. } r.\text{val}[c] \neq v_{\text{del}}[c]$$

Formula Translation Reference

$R_{\text{active}} = \bigcup_{d \in D_k} \Psi(d)$ The final set of active rows in the table is found by going through every single active data file ($D_k$), filtering out its deleted rows using the collapse function ($\Psi$), and uniting the remaining active rows.
$\Psi(d)$ filters using $F_{p, k}(d)$ & $F_{e, k}(d)$ For a given data file, only read rows that pass **both** the position delete checks and the equality delete checks that apply to this file.

3. Depth Analysis: Position vs. Equality Deletes

When executing Merge-on-Read, the write engine can write deletions using either Position Deletes or Equality Deletes. These represent diametrically opposed structural strategies and trade-offs.

A. Position Deletes

A Position Delete File tracks exact coordinates of deleted records. Each row in a position delete file has the following schema: 

struct position_delete {
  100: string file_path;      // The absolute URI of the data file containing the deleted row
  101: long pos;              // The zero-indexed row number within the data file
  102: struct<...> row;      // (Optional) The original deleted row content (useful for CDC)
}
  • Write Path Complexity — High ($O(\text{Read-before-Write})$): To write a position delete, the writing engine (e.g., Spark/Flink) must determine the exact physical file path and row number of the record being deleted. This typically requires querying the table's indices or running a join to locate the records, which incurs read-before-write overhead.
  • Read Path Complexity — Low ($O(N)$ sweep or $O(1)$ set checks): During query execution, the scan engine loads the position deletes associated with a data file $d$ into an in-memory sorted set or a highly optimized bit array (e.g., a **Roaring Bitmap**). Because position deletes map to exact indices, the reader can perform constant-time $O(1)$ membership checks for each row read, or even directly skip deleted blocks, yielding near-optimal scan performance.

B. Equality Deletes

An Equality Delete File specifies deleted rows based on matching column values. Each row in an equality delete file contains values for a subset of the table's schema columns, accompanied by manifest-level metadata: 

135: list equality_ids;  // Field IDs from table schema used to determine row equality
Structural Deconstruction
How does a Field ID List [1] constitute an "Equality Delete" rule?

An equality delete does **not** store an SQL string like "id = 104". Instead, it is structured as a column-matching lookup rule split into two parts: the **Rule Definition** (in manifest metadata) and the **Rule Values** (in the physical Parquet file).

1. The Metadata Rule: equality_ids = [1]
The manifest entry declares which columns are the "equality key fields." In our table schema, Field ID 1 corresponds to the column id. This tells the query engine: "To execute this delete, match rows on their id values."
2. The Storage Schema: The Physical Parquet Delete File
Because the delete rule only requires matching on column ID 1, the writing engine writes a physical Parquet file that **only contains the column for ID 1**. It does not waste space storing any other columns. If we want to delete rows where id = 104, the Parquet file physically contains: 
id (Field ID 1)
--------------
104
3. The Reading Engine Execution
During scan planning, when reading a record from a data file:
  • Extracts the value of the column corresponding to Field ID 1 (the id column). Let's say the row is (id=104, name="David"), so value is 104.
  • Queries the in-memory lookup table of the delete file. Since 104 is found in the delete file, the engine matches:
    row.id == delete_file.id  $\rightarrow$  104 == 104  $\rightarrow$  Logical Match: Row is deleted!

💡 Multi-Column Composite Keys: If equality_ids = [1, 2] (where ID 1 is id and ID 2 is category), the Parquet delete file would contain both columns, and a row is only deleted if both fields match simultaneously (a composite primary-key deletion).

  • Write Path Complexity — Low ($O(1)$ append): The writing engine does not need to locate the original files or positions. It simply writes the keys of the deleted rows to the delete file. This is highly suitable for low-latency streaming ingest (e.g., Flink streaming events directly from Kafka) where reading before writing is unacceptably slow.
  • Read Path Complexity — High ($O(M)$ hash tables & scans): At read time, the reader cannot prune deletes at the row-level in $O(1)$. It must build an in-memory hash join table containing all equality delete key values, and evaluate these predicates against *every single record* read from *all* active data files within the partition. This requires massive memory capacity and incurs a major CPU evaluation penalty, often causing Out-of-Memory (OOM) errors if deletes are uncompacted.

C. Implementation Nuance: Row Ordering & The Streaming Two-Pointer Sweep

While relational SQL theory treats tables and unions as unordered sets, real-world query engines (like Apache Spark and Flink) and the Iceberg Java/Rust reader implementations leverage a vital physical invariant: **Row Order Preservation**.

If the table schema specifies a sort order, the data files are written with records sorted according to those sort keys. When reading these files, preserving this physical ordering is critical to avoid expensive downstream sorting operations in the execution plan. The Iceberg Java library guarantees this through two elegant design patterns:

  • Position Delete Streaming Sweep ($O(N)$ Merge): The Iceberg V2 specification mandates that **position deletes inside the Parquet files must be written sorted** by file_path and pos in ascending order. Since the reader streams data rows sequentially from disk (implicitly sorted by position 0, 1, 2...), the engine does not need to load all position deletes into memory. It maintains a single pointer in the data stream and a single pointer in the position delete stream, executing a **two-pointer streaming sweep (like a merge-join)**. This keeps the execution memory footprint to a minimal $O(1)$ stream buffer while perfectly preserving row order.
  • Vectorized Batch Skipping: In vectorized Parquet readers (e.g. using Apache Arrow or Spark's vectorized reader), position deletes are loaded into an in-memory sorted array or bitmap. Because the offsets are sorted, the reader can completely skip reading entire block groups or pages of the Parquet column file if their offsets match a contiguous run of deleted rows, preserving physical alignment and maximum IO throughput without row-by-row decompression.

4. Sequence Number Applicability Invariants

To prevent a delete from erroneously suppressing rows committed *after* the delete was executed, Iceberg scan planning enforces precise mathematical invariants regarding **Sequence Number Gating**.

Let $\text{Seq}(d)$ be the data sequence number of data file $d$, and let $\text{Seq}(f)$ be the data sequence number of delete file $f$. A delete file $f$ is applied to a data file $d$ if and only if they belong to the same partition and satisfy the following conditions:

Invariant 4.1: Position Delete Applicability Constraint
$$\text{Apply}(f_{\text{pos}} \to d) \iff \text{Seq}(d) \le \text{Seq}(f_{\text{pos}})$$
The Invariant: A position delete file $f_{\text{pos}}$ applies to data file $d$ if and only if $\text{Seq}(d) \le \text{Seq}(f_{\text{pos}})$. This permits a position delete file to invalidate rows in data files committed in the same transaction (where sequence numbers are equal). This is essential for transactions (like MERGE INTO) that write both new data files and corresponding position deletes simultaneously.
Invariant 4.2: Equality Delete Applicability Constraint
$$\text{Apply}(f_{\text{eq}} \to d) \iff \text{Seq}(d) < \text{Seq}(f_{\text{eq}})$$
The Invariant: An equality delete file $f_{\text{eq}}$ applies to data file $d$ if and only if $\text{Seq}(d) < \text{Seq}(f_{\text{eq}})$ (strictly less than). This strictly gates the applicability of equality deletes to older commits. An equality delete does *not* apply to data files added in the same commit transaction. This prevents complex circular lookup dependencies and guarantees that newly appended data rows in a transaction are not immediately suppressed by equality deletes committed concurrently in the same batch.

5. Shannon Entropy Limits & Deletion Vector Roaring Containers

To minimize network round-trips (RTT) and object storage footprints, modern query engines leverage Deletion Vectors. Instead of writing verbose Parquet files for position deletes, deletion vectors serialize the exact row offset bitmaps directly as compressed binary payloads. Under the hood, this relies on deep information theory bounds and the thermodynamic properties of Roaring Bitmaps.

💡 Deletion Vectors: The Plain English Concept
Imagine you have a book with 1,000,000 pages, and you want to tear out 1,000 specific pages.
• Traditional Position Deletes is like publishing a second complete "delete book" that repeats the full title, file path, and exact chapter header for every single torn page. It is incredibly verbose and heavy to read.
• Deletion Vectors (Roaring Bitmaps) is like keeping a tiny index card in the front cover. Instead of writing out full numbers like "Page 987,654 is deleted", it groups pages by blocks of 65,536 (like zip codes). It writes: "In the 987,000s block, tear out page 654." This zip-code grouping allows us to store page numbers as tiny 16-bit offsets instead of massive 64-bit numbers, compressing the list to almost nothing.

A. Mathematical Foundations: Shannon Entropy & Chunk Gating Limits

Under Shannon's information theory, the minimum theoretical bit budget $H$ required to represent $M$ deleted row offsets in a physical data file containing $N$ total rows corresponds to the entropy of choosing $M$ elements from $N$: $$H = \log_2 \binom{N}{M} \text{ bits}$$

In Plain English: If a file has $N$ total rows, there are exactly $\binom{N}{M}$ possible combinations of how $M$ deletes could be distributed. The theoretical minimum number of bits needed to uniquely identify *our specific combination* of deletes is simply the binary logarithm ($\log_2$) of that count. Any file larger than this limit is physically wasting storage space.

Using Stirling's approximation ($\ln(K!) \approx K \ln K - K$), we can expand the combination and compute the binary entropy per deleted index: $$\ln \binom{N}{M} = \ln(N!) - \ln(M!) - \ln((N-M)!)$$ $$H \approx M \log_2\left(\frac{N}{M}\right) + (N - M) \log_2\left(\frac{N}{N - M}\right) \text{ bits}$$

In Plain English: Since calculating combinations for massive datasets (like $\binom{1,000,000}{1,000}$) overflows standard calculators, we use Stirling's formula to simplify the math. This reveals that the necessary bit budget scales proportionally with the logarithmic density of deletions.

When deletions are sparse ($M \ll N$), the entropy budget per deleted index simplifies to: $$\lim_{M/N \to 0} \frac{H}{M} \approx \log_2\left(\frac{N}{M}\right) + \log_2(e) \approx \log_2\left(\frac{N}{M}\right) + 1.44 \text{ bits per index}$$

For instance, in a data file with $N = 1,000,000$ rows and a sparse deletion count $M = 1,000$: $$\frac{H}{M} \approx \log_2(1,000) + 1.44 \approx 9.97 + 1.44 \approx 11.41 \text{ bits per deleted offset}$$

In Plain English: If only a tiny fraction of rows are deleted (e.g., 1,000 out of a million), we mathematically only need **$11.4$ bits of storage** to represent each deleted offset. Storing them as full 64-bit long values in a Parquet file is like parking a bicycle inside a massive 18-wheeler cargo truck—a $5.6 \times$ waste of space.

To approach this limit, Roaring Bitmaps partition the 32-bit integer offset space into logical chunks of size $2^{16} = 65,536$. Every global row offset $x$ is decomposed into a chunk key and a local value: $$x = \text{key} \times 2^{16} + \text{value} \quad \text{where} \quad \text{key} = \lfloor x / 2^{16} \rfloor \quad \text{and} \quad \text{value} = x \bmod 2^{16}$$

In Plain English: Instead of tracking absolute row numbers like `987,654` using massive 64-bit words, we group rows into blocks of 65,536. We record a zip code (chunk key) and a relative offset (local value). This allows us to use tiny 16-bit (2-byte) integers for all positions, matching the theoretical entropy budget beautifully.

Local values within each chunk are stored in one of three dynamically selected containers:

  • Array Container: Used when the number of deleted rows in the chunk is $M_{\text{chunk}} < 4096$. It stores offsets as a sorted list of 16-bit unsigned integers, consuming exactly $16 \times M_{\text{chunk}}$ bits.
  • Bitmap Container: Used when $M_{\text{chunk}} \ge 4096$. It swaps the representation to a dense bit array of $2^{16}$ bits (exactly $8,192$ bytes), capping the space to $\le 16$ bits per offset.
  • Run-Length Encoding (RLE) Container: Used when deletes are contiguous (e.g. bulk drops). It stores runs as 16-bit pairs of (start_offset, run_length).
⚖️ Mathematical Proof: The 4096 Array-to-Bitmap Crossover Point
Why is $M_{\text{chunk}} = 4096$ the exact transition boundary between array and bitmap containers?
Let $M_{\text{chunk}}$ be the number of deleted offsets in a $2^{16}$ range chunk.
• An Array Container stores each offset as a 16-bit (2-byte) integer: $\text{Size}_{\text{Array}} = 2 \times M_{\text{chunk}} \text{ bytes}$.
• A Bitmap Container stores the chunk as a dense bit array covering all 65,536 indices: $\text{Size}_{\text{Bitmap}} = \frac{65,536 \text{ bits}}{8} = 8,192 \text{ bytes}$.
To find where the bitmap container becomes more space-efficient than the array container, we solve: $$\text{Size}_{\text{Array}} > \text{Size}_{\text{Bitmap}} \implies 2 \times M_{\text{chunk}} > 8,192 \implies M_{\text{chunk}} > 4,096$$ Thus, the instant a chunk contains **strictly more than 4,096** deleted offsets, swapping to a dense bitmap container is mathematically guaranteed to reduce memory and storage footprints!

B. Physical Byte-Level Binary Layout on Disk

Under the Iceberg Format V2 specification, Deletion Vectors can be serialized as raw binary blocks inside Puffin files. A serialized Deletion Vector follows a rigid, high-performance byte-level protocol:

D. Complete Vectorized Deletion Vector Reader & Decoder Implementation

The following Python program implements a complete vectorized binary parser and decoder for serialized Deletion Vectors, illustrating how engines like PyIceberg reconcile logical deletes during scanning loops: 

import struct
from typing import List, Dict

class RoaringContainer:
    """Interface for Roaring Bitmap chunk containers."""
    def contains(self, x: int) -> bool:
        raise NotImplementedError()

class ArrayContainer(RoaringContainer):
    """Sorted array container for sparse chunks (M < 4096)."""
    def __init__(self, values: List[int]):
        self.values = sorted(values) # 16-bit sorted integers
        
    def contains(self, x: int) -> bool:
        # Binary search: O(log M) complexity
        low, high = 0, len(self.values) - 1
        while low <= high:
            mid = (low + high) // 2
            if self.values[mid] == x:
                return True
            elif self.values[mid] < x:
                low = mid + 1
            else:
                high = mid - 1
        return False

class BitmapContainer(RoaringContainer):
    """Dense bit-array container for highly deleted chunks (M >= 4096)."""
    def __init__(self, bits: bytearray):
        # Dense bitset representing 65,536 integers (8,192 bytes)
        self.bits = bits
        
    def contains(self, x: int) -> bool:
        # O(1) bit shift check
        byte_idx = x >> 3
        bit_idx = x & 7
        return (self.bits[byte_idx] & (1 << bit_idx)) != 0

class RleContainer(RoaringContainer):
    """Compressed run-length encoded container for contiguous runs."""
    def __init__(self, runs: List[tuple]):
        # List of (start_offset, run_length)
        self.runs = runs
        
    def contains(self, x: int) -> bool:
        # Binary search over sorted contiguous runs: O(log R)
        low, high = 0, len(self.runs) - 1
        while low <= high:
            mid = (low + high) // 2
            start, length = self.runs[mid]
            if start <= x <= start + length:
                return True
            elif x < start:
                high = mid - 1
            else:
                low = mid + 1
        return False

class DeletionVector:
    """
    Parses a serialized binary Deletion Vector and performs 
    vectorized row filtering during physical scan sweeps.
    """
    def __init__(self):
        self.chunks: Dict[int, RoaringContainer] = {}

    @classmethod
    def deserialize(cls, data: bytes) -> 'DeletionVector':
        dv = cls()
        if not data:
            return dv
            
        # Parse standard binary structure:
        # [Magic: 1 Byte] [Num Chunks: 4 Bytes (BE)]
        magic = data[0]
        assert magic == 0x52, "Invalid Magic byte! Expected 0x52 ('R')"
        
        num_chunks = struct.unpack_from(">I", data, 1)[0]
        offset = 5
        
        for _ in range(num_chunks):
            # Parse chunk header: [Key: 2B] [Container Type: 1B] [Payload Size: 2B]
            key, c_type, size = struct.unpack_from(">HBH", data, offset)
            offset += 5
            payload = data[offset : offset + size]
            offset += size
            
            if c_type == 0:  # Array Container
                values = []
                for i in range(0, len(payload), 2):
                    values.append(struct.unpack_from(">H", payload, i)[0])
                dv.chunks[key] = ArrayContainer(values)
                
            elif c_type == 1:  # Bitmap Container
                dv.chunks[key] = BitmapContainer(bytearray(payload))
                
            elif c_type == 2:  # RLE Container
                runs = []
                for i in range(0, len(payload), 4):
                    start, length = struct.unpack_from(">HH", payload, i)
                    runs.append((start, length))
                dv.chunks[key] = RleContainer(runs)
                
        return dv

    def is_deleted(self, row_offset: int) -> bool:
        """
        Evaluate if a physical row position offset is marked as deleted.
        Complexity: O(1) chunk indexing + container lookup (O(1) to O(log M)).
        """
        key = row_offset >> 16      # Extract upper 16 bits
        val = row_offset & 0xFFFF   # Extract lower 16 bits
        
        if key not in self.chunks:
            return False
        return self.chunks[key].contains(val)

    def filter_batch(self, batch_offsets: List[int]) -> List[int]:
        """Vectorized filter keeping only non-deleted row offsets."""
        return [off for off in batch_offsets if not self.is_deleted(off)]
