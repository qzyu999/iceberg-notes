V. Transaction Semantics & Optimistic Concurrency Control (OCC)

Iceberg allows concurrent writers to execute transactional updates without locks, performing validation assertions at the moment of catalog swap.

Let $T_{\text{commit}}$ be a transaction reading table state $S_{\text{base}}$ and proposing mutation updates $\Delta = (D_{\text{add}}, D_{\text{del}}, F_{\text{add\_del}}, F_{\text{rem\_del}})$. Let $S_{\text{current}}$ be the current latest snapshot committed in the catalog. If $S_{\text{current}} \neq S_{\text{base}}$, the catalog validation engine **must** assert the following four safety theorems before permitting the pointer swap:

Invariant 1: Existence Integrity
$$\forall f \in F_{\text{add\_del}}, \quad f.\text{target\_path} \in \{ d.\text{path} \mid d \in D_{\text{current}} \}$$
Invariant 1: New delete files added by our transaction must target data files that are still active in $S_{\text{current}}$. Aborts with Cannot delete rows in missing data file if violated.
Invariant 2: No Double Delete
$$D_{\text{del}} \cap ( D_{\text{base}} \setminus D_{\text{current}} ) = \emptyset$$
Invariant 2: Data files deleted by our transaction must not have been concurrently removed by another transaction. Aborts with File already deleted by concurrent transaction if violated.
Invariant 3: Serializable Append Conflict
$$\forall d \in (D_{\text{current}} \setminus D_{\text{base}}), \quad \text{MatchesPredicate}(d.\text{stats}, \Phi) = \emptyset$$
Invariant 3: Under SERIALIZABLE isolation, no concurrent transaction could have appended data files matching our transaction's delete filter predicate $\Phi$. Aborts with Found conflicting concurrent appends if violated.
Invariant 4: Deletion Vector Integrity
$$\forall v \in V_{\text{add}}, \quad v.\text{target\_path} \notin \{ d.\text{path} \mid d \in (D_{\text{base}} \setminus D_{\text{current}}) \}$$
Invariant 4: No concurrent transaction could have rewritten or compacted a data file targeted by our newly added Deletion Vectors. Prevents bit offsets mapping onto missing data.
Theorem 2: Strict Serializability of Commits
$$\text{Commit}(\Delta, S_{\text{current}}) \equiv \text{SerialExecution}(S_{\text{current}} \to S_{\text{current+1}})$$
Theorem 2: If the transaction validation invariants (Invariants 1-4) evaluate to true at the moment of catalog commit, the concurrent execution of transaction $\Delta$ against $S_{\text{current}}$ is equivalent to a strictly serial execution where $\Delta$ was executed directly on top of $S_{\text{current}}$.
Proof of Theorem 2:
Let $S_{\text{current}}$ represent the state resulting from the serialization of concurrent transactions committed after $S_{\text{base}}$. We must show that applying $\Delta$ to $S_{\text{current}}$ yields a logically equivalent state to a serial execution.
  1. By Invariant 2: Since our deleted files ($D_{\text{del}}$) and concurrently deleted files ($D_{\text{concurrent\_del}}$) are completely disjoint, there are no overlapping file deletions. The physical deletion operations commute perfectly.
  2. By Invariant 1: Since all target data files for our new delete files still exist in $D_{\text{current}}$, no delete records are orphaned.
  3. By Invariant 3: Under serializable isolation, since no concurrently added files match our delete filter predicate $\Phi$, our transaction's deletions would have had zero effect on the concurrent appends. Thus, the operations serialize.
  4. By Invariant 4: Since target data files for our roaring bitmaps were not concurrently deleted or compacted, the bit masks align correctly to the targeted rows.
Because all structural modifications and file changes commute without violating references, the execution is strictly serializable. $\quad \blacksquare$
